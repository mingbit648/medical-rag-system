
# 毕设项目需求文档（PRD）

**项目名称**：基于混合检索策略的法律辅助咨询智能体系统（劳动争议垂直领域）  
**部署形态**：本地单机部署（开发机/服务器均可）  
**定位声明**：用于法律信息检索与学习的辅助系统，不替代正式法律意见或诉讼代理建议。

---

## 1. 需求概述

### 1.1 背景与问题

通用大模型在法律垂直领域（尤其劳动争议）容易出现：知识覆盖不完整、回答缺乏明确依据、可核验性弱（幻觉）等问题。本项目通过**本地权威知识库 + RAG（检索增强生成）+ 混合检索（BM25+向量）+ 重排序（Cross-Encoder）+ 引用溯源**，提升回答的可信度与可追溯性。

### 1.2 核心目标（Goals）

1. 构建可运行的端到端系统：**文档接入 → 清洗切分 → 索引构建 → 混合检索 → rerank → RAG 生成 → 引用溯源展示**。
    
2. 回答必须提供可核验依据：**引用片段与出处可查看**。
    
3. 完成实验评估：使用 ≥20 个劳动争议问题，默认采用 **Recall@5、MRR** 进行对比实验（基线 vs 改进）。
    

### 1.3 用户与使用场景

- **普通用户**：口语化提问劳动争议问题，希望获得“解释 + 引用依据 + 进一步建议（非正式法律意见）”。
    
- **初级法务/律师助理/学生**：希望快速定位相关法条/司法解释/案例要旨，降低检索成本。
    

### 1.4 输入与输出

**输入**

- 知识侧：法规、司法解释、办事指引、典型案例等（PDF/HTML）。
    
- 交互侧：用户自然语言问题（支持多轮对话）。
    

**输出**

- 主输出：生成式回答（可 Markdown 渲染）
    
- 证据输出：引用列表（每条引用含：来源信息 + 引用片段）
    
- 引用查看：打开原文上下文并**高亮**显示被引用片段（P1）。

## 2. 功能需求（P0 / P1 / P2）

> P0：必须交付（核心链路）  
> P1：增强体验/论文展示加分  
> P2：可选优化（不影响毕业基本交付）

### 2.1 P0 功能需求（必须）

#### FR-01 文档接入与解析（PDF/HTML）

- 支持导入 PDF/HTML 文档并解析为可处理文本
    
- 做基础清洗（去除明显页眉页脚/乱码/重复行等），保留可追溯的来源元数据（标题、来源、发布时间如可得、URL/文件名等）
    

#### FR-02 文本分块（Sliding Window Chunking）

- 对长文本按滑动窗口切分，支持重叠窗口
    
- 每个 chunk 必须绑定：chunk_id、doc_id、chunk_text、必要元数据（章节/条款号如可解析）
    

#### FR-03 索引构建：向量索引 + BM25 倒排索引

- 向量索引：存储 chunk embedding（向量库如 ChromaDB/FAISS 均可）
    
- BM25 索引：对 chunk 文本构建倒排索引，支持关键词/条款号精确命中
    
- 两路索引均需返回 chunk_id 与来源元数据
    

#### FR-04 混合检索（Hybrid Search）：双路召回 + 融合

- 并行召回：BM25 TopN + 向量 TopN
    
- 融合：支持 RRF（倒数排名融合）或等价融合策略，输出融合后的候选集合 TopK
    

#### FR-05 重排序（Rerank）：Cross-Encoder

- 对融合候选 TopK 进行 Cross-Encoder 重排序
    
- 输出最终用于生成的 TopM evidence chunks，并保留 rerank 分数（用于实验与调试）
    

#### FR-06 RAG 问答引擎：检索增强生成

- 将 TopM evidence chunks 拼装为上下文，调用大模型（API 方式，模型可配置）生成回答
    
- 生成策略要求：回答以检索证据为主，不得脱离证据做“断言式结论”（至少在提示词/链路上约束）
    

#### FR-07 引用溯源（Source Attribution）：片段级引用（chunk）

- 每次回答必须输出引用列表
    
- 每条引用包含：来源信息（文档名/章节/条款号/URL或文件名等）+ 引用片段（chunk_text 或摘要）
    

#### FR-08 后端服务：FastAPI 接口 + 流式响应

- 提供问答 API：支持请求-响应与流式输出（Streaming）
    
- 提供会话能力：至少支持同一会话多轮上下文（session_id + history）
    
- 提供检索/调试 API（可选但建议）：返回本次检索与 rerank 结果用于实验与论文展示
    

#### FR-09 前端：React 对话界面

- 对话式 UI：输入框、消息气泡
    
- 支持 Markdown 渲染
    
- 支持流式输出展示（打字机效果）
    
- 支持引用列表展示（每条引用可点击查看）
    

#### FR-10 测试集与对比实验（≥20 问）

- 构建不少于 20 个劳动争议典型问题测试集
    
- 基线：仅向量检索（Vector-only）
    
- 实验：混合检索 + rerank
    
- 默认指标：Recall@5、MRR；输出可复现的实验结果表/图
    

---

### 2.2 P1 功能需求（增强）

#### FR-11 引用定位增强（页码/定位信息）

- 若 PDF/HTML 解析可得页码/anchor，则在引用中展示（提升可核验性）
    
- 无法获取时不阻塞 P0（可为空）
    

#### FR-12 检索过程可视化/解释（论文展示友好）

- 展示 BM25 命中（关键词/条款号）、向量相似度、融合排序、rerank 排名（可在调试页/开发模式下显示）
    

#### FR-16 引用查看高亮（你提出的新增点）

- 用户点击引用后，展示该引用的原文上下文，并将被引用 chunk 在上下文中**高亮**显示
    
- 支持自动滚动定位到高亮位置
    
- 高亮定位优先用结构化定位（offset/span）；无法精确定位时提供兜底展示（整体高亮 chunk_text 并提示原因）
    

---

### 2.3 P2 功能需求（可选优化）

#### FR-14 多知识域扩展

- 在劳动争议基础上，支持扩展到其他法律方向（通过配置接入不同语料）
    

#### FR-15 可引用性规则过滤

- 在 rerank 后增加引用质量规则（过短、无出处、疑似噪声片段不允许引用）

## 3. 非功能需求（NFR）

### 3.1 性能与体验

- NFR-01 流式输出：用户提问后需可见流式生成（前端打字机效果）
    
- NFR-02 稳定性：解析失败、模型调用失败、索引缺失等异常场景必须有可理解的错误提示，不应崩溃
    

### 3.2 可维护与可扩展

- NFR-03 管道清晰：文档解析、切分、入库、索引构建、检索、rerank、生成模块解耦
    
- NFR-04 可复现：实验配置、TopK/TopM、融合策略参数、rerank 模型配置需可记录与复现
    

### 3.3 安全与合规表达

- NFR-05 合规声明：UI/README/回答模板中必须明确“仅供学习与辅助检索，不构成法律意见”
    
- NFR-06 可追溯：每条答案必须带引用来源；引用可查看、可核验
    

---

## 4. 边界与约束

### 4.1 明确不做

- 不提供正式法律意见/诉讼策略替代律师
    
- 不保证“百分百正确”，系统强调“给出依据并可核验”
    
- 不做复杂权限、账号体系、付费、运营后台（除非后续新增需求）
    

### 4.2 技术与实现约束（你已确认/项目限定）

- 部署：本地单机
    
- 数据：公开权威渠道，教学/研究用途
    
- 模型：通过 API 调用，具体模型可配置（暂不锁定）
    
- 核心检索链路：必须实现混合检索 + rerank（非仅向量检索）
    

### 4.3 外部依赖

- 权威法律数据来源的可获取性与格式（PDF/HTML 质量差异）
    
- 模型 API 可用性（Key、网络、限流/费用等）

## 5. 验收标准（Given-When-Then）

> 以下为 P0 核心 + P1 高亮的可验证标准（建议直接转测试用例）。

### AC-01 文档接入与解析（FR-01）

- **Given** 系统提供导入 PDF/HTML 的入口或脚本  
    **When** 导入一份法规 PDF 与一份案例 HTML  
    **Then** 输出可机读文本并保留来源元数据，文本可进入切分与索引流程。
    

### AC-02 滑动窗口切分（FR-02）

- **Given** 一段较长法规文本  
    **When** 执行滑动窗口切分（含重叠）  
    **Then** 生成 chunk 集合，chunk 之间存在重叠上下文；抽检不出现关键条款被硬切断导致语义缺失的明显问题。
    

### AC-03 索引构建（FR-03）

- **Given** chunk 集合  
    **When** 分别构建向量索引与 BM25 索引  
    **Then** 两路检索均能返回 chunk_id 与出处元数据，且可按 chunk_id 回溯到 chunk_text。
    

### AC-04 混合检索与融合（FR-04）

- **Given** 同一问题包含术语/条款号与口语化描述两种表达  
    **When** 执行混合检索  
    **Then** 候选集合同时包含 BM25 强命中片段与语义相近片段，并输出融合后的 TopK 列表。
    

### AC-05 重排序（FR-05）

- **Given** 融合候选 TopK  
    **When** 进行 Cross-Encoder rerank  
    **Then** 输出 rerank 后 TopM evidence，且可观察到与问题不相关的“噪声片段”排名下降（在测试集中可复现）。
    

### AC-06 RAG 生成回答（FR-06）

- **Given** 用户提交劳动争议问题  
    **When** 系统完成检索、rerank、上下文拼装并调用模型 API  
    **Then** 返回结构清晰的回答文本，且回答内容与引用证据语义一致，不出现完全无依据的断言。
    

### AC-07 引用溯源展示（FR-07）

- **Given** 系统生成回答  
    **When** 前端展示该回答  
    **Then** 必须展示引用列表；每条引用包含来源信息与引用片段（chunk）。
    

### AC-08 流式响应与会话（FR-08/FR-09）

- **Given** 用户开始一次会话  
    **When** 提问并等待生成  
    **Then** 前端以流式方式逐字展示回答；同一 session 再次提问时，系统能维持多轮对话上下文。
    

### AC-09 对比实验（FR-10）

- **Given** ≥20 个劳动争议测试问题集  
    **When** 分别运行基线（仅向量）与实验（混合+rerank）  
    **Then** 输出 Recall@5、MRR 的结果对比表/图，并给出结论（实验组优于或至少不劣于基线，且解释原因）。
    

### AC-16 引用查看高亮（FR-16，P1）

- **Given** 某次回答包含引用列表，引用记录包含 chunk_id 与 chunk_text  
    **When** 用户点击任意引用的“查看原文”  
    **Then** 展示包含该引用的上下文内容，并将 chunk_text 对应区域高亮，且自动定位到高亮位置。
    

### AC-16b 高亮失败兜底（鲁棒性）

- **Given** 由于解析差异导致 chunk_text 无法在上下文中精确匹配  
    **When** 用户查看引用  
    **Then** 系统仍展示引用内容：至少展示 chunk_text 并整体高亮，同时提示“原文排版差异导致无法精确定位”，不影响用户核验。
