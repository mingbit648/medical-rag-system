## 2. 项目定位

### 2.1 项目名称

基于混合检索策略的法律辅助咨询智能体系统

### 2.2 业务场景

本项目面向劳动争议为主的法律咨询场景，核心目标不是替代律师，而是提供：

- 面向普通用户的法律信息辅助检索与问答
- 面向初级法务/律师助理的法条、案例片段辅助定位
- 面向论文答辩与实验对比的混合检索效果验证能力

### 2.3 项目边界

系统当前边界比较明确：

- 做“辅助咨询”和“依据检索”
- 做“引用溯源”和“原文定位”
- 做“实验评测接口”
- 不做正式法律意见
- 不做生产级鉴权与多租户
- 不做复杂工作流编排或案件管理系统

### 2.4 开题报告中的研究目标

开题报告希望解决三个核心问题：

1. 通用大模型在法律场景下的幻觉问题
2. 法律文本长文本切分后语义断裂的问题
3. 回答结果缺少可核验依据的问题

当前仓库确实围绕这三个点搭了主链路，但落地程度不完全一致，后文会把“已实现”和“仍有差距”的地方拆开说清楚。

## 5. 后端架构设计

### 5.1 启动入口

`backend/app/main.py`

职责：

- 创建 FastAPI 应用
- 注册 CORS
- 在 lifespan 中初始化 `get_engine()`
- 挂载 docs/chat/retrieve/citations/experiments 路由
- 暴露 `/health`

### 5.2 路由层设计

| 路由模块 | 主要职责 |
| --- | --- |
| `routers/docs.py` | 文档导入、建索引、状态查询、原文下载、viewer 内容获取、删除 |
| `routers/chat.py` | 创建会话、会话列表、消息列表、普通问答、流式问答、删除会话 |
| `routers/retrieve.py` | 检索调试接口 |
| `routers/citations.py` | 引用查看、原文打开目标 |
| `routers/experiments.py` | 评测运行、评测记录查询 |

### 5.3 核心服务分层

#### 5.3.1 `rag_engine.py`

这是系统门面层（facade），负责编排：

- 文档导入
- 索引重建
- 检索
- 生成
- 会话收尾
- 引用查看
- 实验评测

它本身不把所有逻辑糊成一坨，而是把能力拆到多个 `core` 模块里，这点是对的。

#### 5.3.2 `doc_ingestion.py`

职责：

- 识别上传文档类型（PDF / DOCX / HTML / TXT）
- 抽取纯文本
- 生成页码或段落位置信息
- 保存原始文件到 `data/uploads/<doc_id>/original.*`
- 把结构化元信息与正文写入数据库

工程价值：

- 后续引用不只展示 snippet，还能反查原件
- PDF 和 DOCX 的定位元数据被保留下来

#### 5.3.3 `chunker.py`

当前实现是：

- 固定窗口大小 `chunk_size`
- 重叠窗口 `overlap`
- 为 chunk 生成 `start_pos/end_pos`
- 尝试提取 `section/article_no`
- 结合 `page_spans/paragraph_spans` 生成 `locator_json`

优点：

- 简单稳定
- 方便定位引用
- 已能支撑 viewer 高亮

局限：

- 还是偏字符级切分
- 没有针对法条、款、项、案例事实段落做专门边界识别

#### 5.3.4 `embedder.py`

Embedding 路径有三层回退：

1. SiliconFlow 在线 embedding
2. 本地 `SentenceTransformer`
3. Hash embedding

这套设计很务实，优点是系统不容易因为模型依赖没装齐就彻底起不来。

#### 5.3.5 `retriever.py`

检索流程：

1. BM25 关键词召回
2. 向量召回
3. RRF 融合
4. rerank

rerank 支持两条路：

- SiliconFlow rerank API
- 本地 CrossEncoder

如果都不可用，则回退到启发式分数（词项重叠 + 向量分 + BM25 归一化）。

#### 5.3.6 `generator.py`

职责：

- 组装证据文本
- 构造 prompt
- 调用 OpenAI 兼容接口的 LLM
- 流式生成
- 构造 citation payload

当前生成策略的核心原则是：

- 命中证据时优先基于证据作答
- 没命中时给保守分析
- 末尾统一追加免责声明

#### 5.3.7 `services/session_service.py`

这是比较像工程代码的地方，做了不少状态控制：

- 新会话创建
- `request_id` 去重
- 正在流式生成中的会话写保护
- 消息顺序号 `session_seq`
- 上下文摘要 `session_context_snapshots`
- stale streaming message 修复
- 会话标题自动生成

它不是“为了有会话而有会话”，而是真在控制会话生命周期。

### 5.4 数据访问层

`repositories/pg_repository.py`

特点：

- 用 `SimpleConnectionPool`
- 全手写 SQL
- 启动时自动 `ALTER TABLE IF NOT EXISTS`
- 同时补索引、补约束、补历史兼容字段

优点：

- 没 ORM 包袱
- 对当前体量来说直给、可控

问题：

- 把 schema 演进逻辑塞进运行时，适合毕业设计，不适合长期生产维护
- 项目虽然安装了 `sqlalchemy/asyncpg/alembic`，但实际没走那套，技术栈声明和实现脱节

## 6. 数据模型设计

### 6.1 主要表

| 表名 | 用途 |
| --- | --- |
| `documents` | 文档主表，保存标题、类型、原文路径、全文文本、解析状态、元信息 |
| `chunks` | 文档分块表，保存 chunk 内容、位置、节/条、定位元数据 |
| `sessions` | 会话主表 |
| `messages` | 会话消息表 |
| `session_context_snapshots` | 会话摘要快照，给长对话做上下文压缩 |
| `citations` | 引用表，把回答和知识片段绑定 |
| `runs` | 实验评测结果表 |
| `indices_state` | 索引状态表，当前存在但实际使用度不高 |

### 6.2 关键设计点

- `documents.content_text` 保存正文，避免把全文塞进 `meta_json`
- `chunks.locator_json` 负责原文定位
- `messages.status` 区分 `streaming/completed/error`
- `citations.message_id` 让引用和具体回答绑定
- `session_context_snapshots` 避免长对话直接把全部消息塞给模型

### 6.3 当前数据设计评价

优点：

- 结构比普通 demo 完整得多
- 能支撑溯源与历史会话

不足：

- 还没有真正的文档版本管理
- 还没有知识库来源可信级别、发布时间、法效层级等法律元数据
- 还没有增量重建与局部重建机制

## 8. 核心业务流程

### 8.1 文档导入与建索引流程

1. 前端上传文件到 `/api/v1/docs/import`
2. 后端识别类型并抽取文本
3. 保存原始文件到 `data/uploads/<doc_id>/`
4. `documents` 落库
5. 调用 `/api/v1/docs/{doc_id}/index`
6. 按 chunk 规则切分
7. 写入 `chunks`
8. 重建 BM25 索引与向量索引
9. 若启用 Chroma，则同步 upsert 到 Chroma collection

### 8.2 问答流程

1. 用户发送问题
2. `SessionService.start_turn()` 创建 user/assistant 两条消息
3. 基于会话摘要 + 最近消息构建 retrieval query
4. 执行 BM25 和 dense retrieval
5. RRF 融合
6. rerank 排序
7. 依据阈值选取 evidence
8. 生成 citation 并落库
9. 组装 prompt 调 LLM
10. 写回 assistant 内容、状态、debug 信息

### 8.3 流式回答流程

前端通过 SSE 消费：

- `metadata`
- `token`
- `done`
- `error`

这比“后端一次性返回整段文本”强，用户体验上已经接近可演示产品。

### 8.4 引用查看流程

1. 回答中保留 citation 列表
2. 前端点击引用
3. 调 `/api/v1/citations/{citation_id}/view`
4. 若原件可用，再调用 `/open-target`
5. PDF 直接跳原始文件页码
6. 文本类文档进入 `/document-viewer/` 高亮展示

### 8.5 实验评测流程

1. 脚本准备测试集
2. 调 `/api/v1/experiments/run`
3. 计算 baseline vector-only 与 hybrid+rerank 指标
4. 指标写入 `runs`

这条链路已经存在，但从现有脚本和结果文件看，实验数据产物管理还不算成熟。

## 11. 项目现状分析

### 11.1 当前已经做出来的东西

这项目最起码已经具备一个完整可演示闭环：

- 上传法律文档
- 建索引
- 会话式提问
- 混合检索
- 生成带引用的回答
- 查看引用上下文
- 打开原文定位
- 保存会话和实验结果

这已经超过一般毕业设计里“能问一句、回一段废话”的水平。

### 11.2 当前最主要的问题

#### 1）研究方案与实际实现存在偏差

- 开题报告写了 LangChain，但仓库实际没用
- 开题报告把 Chroma 说得像核心组件，实际默认运行是 memory 向量路径

这不一定是错，但论文和答辩里必须说清楚：你最终实现采取了“轻量手写编排 + 可选 Chroma”的方案，而不是照着开题报告逐字复刻。

#### 2）切分策略还不够“法律语义化”

当前 `chunker.py` 本质仍是“固定长度字符切块 + overlap”，只是附带位置元数据。
这能跑，但还不够体现“针对法律文本结构优化”的研究价值。

#### 3）质量依赖外部模型与环境配置

- LLM 质量依赖 DeepSeek 或兼容接口
- embedding/rerank 质量依赖 SiliconFlow 或本地 ML 依赖
- 没 key 时会回退 hash embedding 和启发式 rerank

工程上这是容灾，论文上这就是要讲清楚的“实验环境差异变量”。

#### 4）仓库内存在编码不一致/乱码问题

从多个文件读取结果看，README 与开题报告是正常中文，但 `main.py`、`config.py`、`layout.tsx`、`text_utils.py`、`chunker.py`、部分页面文本等文件存在明显乱码痕迹。
这说明仓库编码状态不统一，后续不处理，展示效果和维护成本都会受影响。

#### 5）配置与文档漂移

- README 仍强调 `/legal-shell`
- 实际首页跳转到 `/chat/`
- 前端开发默认 API 指向 `8002`
- 后端本地启动文档是 `8000`
- Docker 映射后对外是 `8001`

这套配置现在是能把人绕晕的。

#### 6）数据库演进方式不够规范

目前通过运行时 `ALTER TABLE IF NOT EXISTS` 来补 schema。
毕业设计能用，长期维护不行。因为：

- 版本演进不可追踪
- 不利于回滚
- 多环境一致性不好管

#### 7）观测性不足

目前系统有 logger，但缺少：

- 结构化日志
- 请求级 tracing
- 检索耗时统计
- LLM 调用耗时与失败率统计
- 引用命中率、空召回率等业务指标

没有这些，优化就容易变成拍脑袋。

#### 8）安全能力明显不足

当前系统没有：

- 登录鉴权
- RBAC
- 上传内容安全校验
- 限流
- 审计日志

如果只是毕业设计演示还能说得过去，真部署到公网就是裸奔。

#### 9）实验链路还不够扎实

仓库里虽然有实验脚本、数据集、`runs` 表和评测接口，但现有结果产物管理比较乱，`experiment_results.json` 当前并不是一份干净的实验结果文件。
这说明实验流程已经搭了，但“实验复现实证”还没真正收干净。

### 11.3 当前阶段结论

一句话总结：
这是一个已经打通主链路、具备论文工程原型价值的法律 RAG 系统，但距离“结构稳定、实验扎实、文档统一、可答辩可维护”的状态，还差一轮系统性收口。

## 14. 对论文撰写与答辩的建议

### 14.1 论文中必须如实说明的点

- 最终实现没有采用 LangChain，而是选择了可控的手写编排
- Chroma 是可选向量后端，不是默认主通路
- 默认可运行模式与高质量模型模式存在差异
- 当前 chunking 已实现 overlap，但尚未完全达到法律语义切分的最终目标

### 14.2 答辩时建议重点强调

- 系统不是只做问答，而是做了“可追溯引用 + 原文定位”
- 混合检索 + rerank 的价值可以通过实验指标证明
- 会话上下文摘要说明系统考虑了长对话场景
- 工程实现能 Docker 部署，不是只会在 IDE 里截图

### 14.3 不建议硬吹的点

- 不要把当前 chunking 说成“完全基于法律语义”
- 不要把可选 Chroma 说成已经全面稳定落地
- 不要把当前系统说成生产可上线法律服务平台

实话实说反而更稳。

## 15. 最终结论

这个项目当前最准确的定位是：

> 一个围绕劳动争议法律咨询场景构建的、具备文档导入、混合检索、流式回答、引用溯源、原文定位、会话管理和实验评测能力的法律 RAG 工程原型。

它的优点是主链路完整、工程意识明显、已经超出普通课程作业水平；
它的短板是法律语义化切分、配置统一、编码质量、实验闭环和生产级工程能力还没完全补齐。

如果后续按“先补底盘、再补研究、最后补运维”的顺序推进，这项目是完全能撑住毕业设计、论文写作和系统演示的。别先上花活，先把根基按死，项目就能站住。
