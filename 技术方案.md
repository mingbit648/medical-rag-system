
## 1. 技术方案概述

### 1.1 目标与交付物

实现端到端链路：**文档接入/解析 → 清洗切分 → 双索引（BM25+向量）→ 混合检索融合（RRF）→ Cross-Encoder 重排序 → RAG 生成（API 可配置）→ chunk 级引用溯源展示 → 引用“查看原文高亮”**。

### 1.2 关键非功能诉求

- **流式输出与稳定错误提示**（模型/解析/索引异常不崩溃）。
    
- **可复现实验**（Recall@5、MRR；参数可记录）。
    
- **合规声明与可追溯引用**（“仅供学习与辅助检索，不构成法律意见”）。

## 2. 技术选型表

|领域|选型（推荐）|备选|选择理由（面向本地单机/毕业交付）|
|---|---|---|---|
|后端框架|**Python + FastAPI**|Flask / Django|你明确要求 FastAPI + Streaming；异步与 SSE/WebSocket 生态成熟。|
|前端|**React + Vite**|Next.js|轻量、开发快；易实现流式打字机与引用面板。|
|文档解析 PDF|**PyMuPDF(fitz)**|pdfplumber|PyMuPDF 更易拿到 page、text blocks，便于后续高亮定位（bbox/页码）。|
|文档解析 HTML|**readability-lxml + bs4**|trafilatura|快速抽取正文；保留 URL/标题等元数据。|
|分块|**滑窗 chunking（可配置）**|语义分段|PRD 指定 Sliding Window、重叠窗口、chunk 元数据必备。|
|向量嵌入|**Sentence-Transformers（本地 embedding）**|Embedding API（可配置）|本地更可复现；但保留 API 可切换（与你“模型 API 可配置”一致）。|
|向量库|**FAISS（本地）**|ChromaDB|FAISS 单机性能好、依赖少；Chroma 更方便存储/管理。|
|BM25|**Tantivy（Python binding）**|Elasticsearch / Whoosh|Tantivy 单机轻量、检索质量好；ES 过重；Whoosh 偏旧。|
|融合策略|**RRF**|归一化加权|PRD 推荐 RRF；实现简单稳定。|
|重排序|**Cross-Encoder（sentence-transformers）**|bge-reranker / API rerank|PRD 指定 Cross-Encoder rerank 并保留分数用于实验。|
|元数据/会话存储|**SQLite + SQLModel**|Postgres|单机部署最省事；结构化存档满足可复现与审计。|
|任务队列（可选）|本地后台线程/进程池|Celery/Redis|单机导入不大时无需引入 Redis；后续扩展再加。|

## 3. 架构设计说明

### 3.1 分层架构

**表现层（React）**

- Chat 对话、流式输出、引用列表、引用查看（原文/高亮）、实验结果页（表/图）。
    

**业务层（FastAPI）**

- Ingestion：导入、解析、清洗、chunk、入库、建索引
    
- Retrieval：BM25、向量召回、RRF 融合、rerank
    
- Generation：RAG prompt 组装、LLM API 调用、流式转发
    
- Attribution：引用结构化、可回溯
    
- Experiment：离线评测、Recall@5、MRR、导出报告
    
- Observability：日志、trace_id、参数记录
    

**数据层**

- SQLite：文档元数据、chunk、引用定位信息、会话、实验记录
    
- FAISS：向量索引
    
- Tantivy：BM25 索引
    
- 文件系统：原始 PDF/HTML、解析中间产物（可选缓存）
    

### 3.2 核心数据流（时序）

1. **导入**：上传 PDF/HTML → 解析 text + 元数据（doc_id）
    
2. **清洗切分**：清洗 → 滑窗 chunking（chunk_id、doc_id、chunk_text、章节/条款号等）
    
3. **建索引**：embedding → FAISS；chunk_text → Tantivy(BM25)
    
4. **问答**：用户 query → BM25 TopN + 向量 TopN → RRF TopK → Cross-Encoder rerank TopM
    
5. **生成**：TopM 拼装上下文 → LLM API → 流式输出
    
6. **引用**：输出引用列表（chunk 级，含来源+片段）
    
7. **查看原文高亮**：点击引用 → 返回上下文 + 定位信息 → 前端高亮并滚动定位；失败兜底整体高亮 chunk_text

## 5. 接口设计（前后端/模块间契约）

### 5.1 通用约定

- Base URL：`/api/v1`
    
- 认证：P0 可不做（本地单机）；预留 `X-API-Key`（可选）
    
- 响应 envelope：
    

```json
{ "code": 0, "message": "ok", "data": {...}, "trace_id": "..." }
```

### 5.2 错误码（示例）

|code|含义|
|---|---|
|0|success|
|40001|参数非法|
|40401|doc/chunk 不存在|
|40901|索引未就绪|
|50001|文档解析失败|
|50002|embedding 失败|
|50003|LLM 调用失败/超时|
|50004|rerank 失败|
|50005|高亮定位失败（将触发兜底返回）|

### 5.3 文档导入与索引

#### 1) 上传导入（本地文件）

- `POST /docs/import`
    
- `multipart/form-data`: `file`, `source_url?(optional)`, `doc_type: pdf|html`
    
- 返回：
    

```json
{
  "doc_id": "doc_xxx",
  "title": "…",
  "status": "imported"
}
```

#### 2) 构建索引（可同步/异步）

- `POST /docs/{doc_id}/index`
    
- body：
    

```json
{
  "chunk": { "size": 800, "overlap": 200 },
  "bm25": { "enabled": true },
  "vector": { "enabled": true, "embed_model": "bge-m3" }
}
```

- 返回：`{ "status": "indexed", "chunks": 1234 }`
    

#### 3) 索引状态

- `GET /docs/{doc_id}/status`
    
- 返回：解析、切分、索引进度与错误信息。
    

### 5.4 问答与流式

#### 1) 非流式

- `POST /chat/completions`
    
- body：
    

```json
{
  "session_id": "s_123",
  "query": "公司拖欠工资怎么办？",
  "topn": { "bm25": 50, "vector": 50 },
  "fusion": { "method": "rrf", "k": 60 },
  "rerank": { "topk": 30, "topm": 8 },
  "llm": { "provider": "openai_compatible", "model": "xxx", "temperature": 0.2 }
}
```

- 返回（核心：answer + citations）：
    

```json
{
  "answer_md": "...",
  "citations": [
    {
      "citation_id": "c_1",
      "chunk_id": "chk_789",
      "doc_id": "doc_1",
      "source": { "title": "劳动合同法", "url_or_file": "xxx.pdf" },
      "location": { "page": 12, "section": "第X章", "article_no": "第Y条" },
      "snippet": "……",
      "scores": { "bm25": 10.2, "vector": 0.71, "rrf": 0.032, "rerank": 8.91 }
    }
  ]
}
```

> 引用必须是 chunk 级，并带来源与片段。

#### 2) 流式（SSE 推荐）

- `POST /chat/completions:stream`
    
- SSE 事件类型：
    
    - `token`: 增量文本
        
    - `done`: 完成（附 citations）
        
    - `error`: 错误（code/message/trace_id）
        

### 5.5 检索/调试（论文展示友好）

- `POST /retrieve/debug`
    
- 返回 BM25 命中、向量相似度、融合排序、rerank 排名（可直接喂给前端可视化）。
    

### 5.6 引用查看与高亮

#### 1) 获取引用原文上下文（含高亮定位信息）

- `GET /citations/{citation_id}/view`
    
- query：`context_before=400&context_after=400`
    
- 返回：
    

```json
{
  "doc_id": "doc_1",
  "doc_type": "pdf",
  "context_text": "……前文……【命中chunk】……后文……",
  "highlight": {
    "method": "offset",
    "start": 5120,
    "end": 5488
  },
  "fallback": null
}
```

#### 2) 高亮失败兜底返回（必须）

当无法精确匹配时：

```json
{
  "context_text": "...(尽可能的上下文)...",
  "highlight": {
    "method": "whole_chunk",
    "chunk_text": "……",
    "reason": "layout_diff_no_exact_match"
  }
}
```

> 这对应你要求的“优先 offset/span，失败整体高亮并提示原因”。

## 7. 关键设计决策

### 7.1 BM25 与向量双索引：为何不只用向量？

- PRD 明确要求：**混合检索（BM25+向量）+ 融合 + rerank**，用于覆盖“条款号/术语精确命中 + 口语化语义相近”。
    

### 7.2 融合选 RRF（而非加权）

- RRF 对不同打分尺度更鲁棒，参数少、复现容易；适合作为毕业论文“稳定 baseline”。
    

### 7.3 Rerank 放在融合后 TopK

- 计算成本与效果平衡：先用廉价召回扩大覆盖，再用 Cross-Encoder 精排，且保留 rerank 分数用于论文展示与调试。
    

### 7.4 高亮定位策略：offset 优先，PDF 尽量 bbox

- **HTML**：天然 DOM，可用字符 offset/anchor 高亮。
    
- **PDF**：文本抽取与排版易漂移：优先存储“page + text span offset”；若用 PyMuPDF 可进一步存储 bbox（更像真正“在 PDF 上画高亮”）。
    
- **兜底**：无法精确定位时“整体高亮 chunk_text + 提示原因”，保证可核验不中断。
    

### 7.5 LLM/Embedding 模型“可配置”的实现方式

- 抽象 `LLMProvider` 与 `EmbeddingProvider`：
    
    - `openai_compatible`（任意兼容 OpenAI API 的服务）
        
    - `local`（可选：本地模型/本地 embedding）
        
- 配置落到 `config.yaml` + DB 记录（用于实验可复现）。
